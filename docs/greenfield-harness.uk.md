# Greenfield-проєкт з нуля — покрокова інструкція (harness-first)

Сценарій: ти власник проєкту, коду ще немає (по суті — MebleSaaS, приклади знову з нього). Головна перевага greenfield: **harness будується ДО коду, а не наздоганяє його.** Головна небезпека: агенти генерують код швидше, ніж ти встигаєш його розуміти — без гейтів з першого комміту через місяць отримаєш чужу кодову базу у власному репо.

Три інверсії відносно попередніх сценаріїв:
1. CLAUDE.md пишеться не «проти помилок» (perception gap ще нема від чого міряти) — він пишеться **з архітектурних рішень**. Кожне рішення = рядок CLAUDE.md у той самий день.
2. Spec-driven не «впроваджується» — він єдиний спосіб роботи з фічі №1. Директорія specs/ = документація проєкту, безкоштовно.
3. Evals не пишуться окремо — **кожна зроблена фіча стає трейсом**. Через 20 фіч у тебе 20 трейсів самі собою.

## ДЕНЬ 0: рішення → скелет → harness (код ще не пишемо)

### 0.1 — Зафіксуй рішення письмово

`docs/decisions.md` — легкий ADR-формат, по 3-5 рядків на рішення:

```markdown
# 001: Go + chi + pgx/sqlc + goose
Чому: основний стек, high-load досвід, генерація коду з SQL.
Відкинуто: Rails (вакансійний стек ≠ продуктовий вибір тут), Fiber (менша екосистема).

# 002: PostgreSQL, params виробів у JSONB
Чому: гнучкі параметри різних типів меблів без міграції на кожен тип.
Ціна: валідація схеми params — у коді (internal/details/params.go), не в БД.

# 003: Railway для старту
Чому: деплой за годину, PostgreSQL з коробки. Міграція на AWS — коли буде трафік.

# 004: monorepo (Go бекенд + React фронт в одному репо)
# 005: multi-tenant з дня 0 — company_id у кожній tenant-таблиці
```

Це не бюрократія: decisions.md — це сировина для CLAUDE.md і відповідь агенту на «а чому тут так», яку інакше він вигадає сам.

### 0.2 — Скелет репо

```bash
mkdir meblesaas && cd meblesaas && git init
mkdir -p cmd/api internal db/{migrations,queries} web docs specs evals/traces .claude/{skills,agents,commands,hooks}
go mod init github.com/rh/meblesaas
```

### 0.3 — CLAUDE.md v0 (з рішень, не з помилок)

Транслюй decisions.md у робочі правила. Приклад стартового (він буде рости з кожною фічею):

```markdown
# MebleSaaS
B2B SaaS для меблевих компаній: параметричний конструктор → список деталей,
AI-рендер, карта розкрою. Два view: продавець і тех.відділ.

## Stack (рішення в docs/decisions.md — читай перед архітектурними змінами)
Go 1.23 · chi · pgx/v5 · sqlc · goose · PostgreSQL 16 · React+TS+Three.js · Railway

## Структура
- cmd/api — entry point, тонкий
- internal/<domain>/ — вся логіка (details, cutting, render, tenant, ...)
- internal/db — ТІЛЬКИ згенерований sqlc код, руками не редагувати
- db/queries/*.sql → sqlc generate; db/migrations → goose (up І down)
- specs/<feature>/ — spec.md + plan.md кожної фічі. Це документація проєкту.

## Незламні правила (з дня 0)
- Кожна tenant-таблиця має company_id; кожен запит — company_id у WHERE
- Errors: fmt.Errorf("<pkg>: <op>: %w", err)
- Handler = парсинг + виклик сервісу + маппінг помилки. Логіка в internal/<domain>
- Нова фіча починається зі specs/<name>/spec.md — БЕЗ спеки код не пишемо

## Команди
- go test ./... · golangci-lint run · sqlc generate · goose -dir db/migrations up
```

### 0.4 — Hooks з комміту №1

Скопіюй з playbook усі три (`fmt.sh`, `guard.sh`, `tests-green.sh`) у `.claude/hooks/`, конфіг у `.claude/settings.json`. На greenfield це найдешевший момент їх поставити: нуль legacy-шуму, нуль false-positives. Один guard додай специфічний:

```bash
echo "$CMD" | grep -qE 'goose.*down' && deny "goose down тільки руками — щоб не втратити локальні дані мовчки"
```

### 0.5 — Slash-команди `/spec`, `/plan`, `/commit`

Ті самі з playbook, без змін. Плюс одна нова, greenfield-специфічна — `.claude/commands/retro.md`:

```markdown
---
description: Retro після мержа фічі — оновлює harness
---
Фіча $ARGUMENTS змержена. Зроби:
1. Порівняй specs/$ARGUMENTS/spec.md з фінальною реалізацією — випиши розбіжності
2. Для кожної: це діра в шаблоні spec, у плані, чи в CLAUDE.md? Запропонуй правку
3. Створи evals/traces/NNN-$ARGUMENTS.md: prompt = суть фічі одним абзацом,
   checks = 3-4 перевірювані критерії з acceptance criteria
4. Якщо в реалізації з'явилась нова конвенція — запропонуй рядок у CLAUDE.md
Покажи все мені на затвердження, нічого не коммить сам.
```

Оце і є механізм «harness росте разом з кодом» — одна команда після кожного мержа.

Перший комміт: `chore: project skeleton + harness (CLAUDE.md, hooks, commands)`. **Harness закоммічений раніше за перший рядок продуктового коду** — це і є harness-first.

## ДЕНЬ 1–2: walking skeleton (фіча №0)

Перша «фіча» — не конструктор і не рендер, а **наскрізний трасер**: найтонший шматок системи, що проходить усі шари й деплоїться.

### 1.1 — `/spec walking-skeleton`

Scope для MebleSaaS: `POST /projects` + `GET /projects/{id}` (тільки client_name), таблиці companies+projects з міграцією, tenant-middleware (поки company_id з заголовка — auth у non-scope), health endpoint, CI (test+lint), деплой на Railway. Acceptance: curl на прод-URL створює і читає проєкт; запит з чужим company_id → 404.

Non-scope тут найважливіший розділ у житті проєкту: конструктор, auth, фронт, рендер — усе НІ. Скелет має бути нудним.

### 1.2 — `/plan` → implement → verify

Повний цикл з playbook, включно з CI-файлом (звичайний test+lint workflow; plan-verifier з playbook додаси сюди ж — на greenfield він з першого PR, і жоден PR ніколи не існував без плану).

### 1.3 — Критерій готовності дня 2

Продуктова цінність нульова, зате: деплой-пайплайн працює, міграції ганяються, tenant-патерн закладений, CI зелений, перший трейс в evals/. Усі наступні фічі — це «додати м'ясо до скелета», а не «зібрати систему».

## ТИЖДЕНЬ 1–2: вертикальні зрізи + ріст harness

### 2.1 — Фічі тільки вертикальними зрізами

Кожна фіча = шматок цінності через усі шари, а не «зробимо всю БД, потім весь API». Порядок для MebleSaaS (з MVP-фаз): `params-model` (схема параметрів шафи + валідація) → `details-generation` (params → список деталей, серце продукту) → `materials-catalog` → перший екран конструктора. Кожна — через `/spec → /plan → implement → verify → /retro`.

### 2.2 — Правило проти головного ризику greenfield

Агент на порожньому репо радісно згенерує тобі «на виріст»: репозиторій-патерн з інтерфейсами на все, конфіг-систему, дженерик-хелпери. Додай у CLAUDE.md одразу:

```markdown
## YAGNI-гейт
- Жодних абстракцій «на майбутнє»: інтерфейс з'являється при ДРУГІЙ реалізації
- Жодних пакетів utils/, common/, helpers/
- Кожен доданий файл має бути потрібен ПОТОЧНІЙ спеці. Зайве — reviewer ріже
```

І перевіряй це на verify: «покажи файли diff, яких не вимагає жоден крок плану».

### 2.3 — Skills народжуються з retro, не наперед

Не пиши skills у день 0 — нема з чого. Після 2-3 фіч /retro сам почне повторювати одні й ті самі правки → це і є момент виносу в skill. Типово до кінця тижня 2 з'являться ті самі три: `go-testing`, `code-review`, `db-migrations` — але наповнені ТВОЇМИ реальними кейсами, а не теорією.

### 2.4 — Метрики

metrics.md з фічі №0. На greenfield додай колонку **LOC diff** — стеж за співвідношенням $/фічу до розміру фічі. Якщо вартість росте від фічі до фічі при однаковому розмірі — контекст засмічується, CLAUDE.md розповз, або YAGNI-гейт протікає.

## ТИЖДЕНЬ 3: субагенти + evals у бойовому режимі

### 3.1 — Субагенти вмикай, коли кодова база переросла «одну голову»

Ознака: researcher стає корисним (є що досліджувати, ~поява 15-20 файлів у internal/). Заводь ту саму п'ятірку з playbook. Dispatch matrix на greenfield проста: нова фіча → повний конвеєр; правки по свіжому коду → моноліт-сесія (код ще «гарячий» у твоїй голові, оверхед агентів не окупається).

### 3.2 — Evals уже накопичились самі

Після ~6-8 фіч у evals/traces/ лежить 6-8 трейсів від /retro. Напиши run.sh з playbook і зроби перший baseline-прогін. З цього моменту: **зміна CLAUDE.md або skill без прогону evals = зміна коду без тестів.**

### 3.3 — MCP — тільки коли з'явився перший зовнішній стан

Для MebleSaaS це момент інтеграції Replicate (рендер) або наповнення довідника матеріалів. Тоді — meble-materials сервер з playbook (~50 рядків). До того MCP не потрібен: не плоди tool definitions у контексті заради краси. Token-аудит (`/context`) зроби одразу після підключення першого сервера — зафіксуй базову ціну.

## ТИЖДЕНЬ 4+: масштаб, коли він справжній

### 4.1 — Fan-out ×3 — на перших механічних міграціях

Перший реальний привід виникне сам: «додати поле X у всі типи деталей», «перевести всі handlers на новий middleware». Playbook + worktrees з попереднього документа без змін. До появи такого приводу fan-out не форсуй.

### 4.2 — Plugin — коли з'явиться друга людина або другий репо

Пакуй `.claude/` у plugin тоді, коли harness треба перенести: найнятий фронтендер, виділений web/ в окремий репо, або новий side-проєкт (harness з MebleSaaS на 80% перевикористовний). Раніше — це папка в git, плагін нічого не додає.

### 4.3 — CI-агенти

Auto-review workflow з playbook став сенс мати з тижня 2 (він же працює і коли PR робить агент — другий агент з reviewer-skill ловить те, що пропустив перший). DevDigest-аналог для соло-проєкту: щотижневий агентський звіт «що змержено / що в specs/ не почато / де тести найтонші» — дешевий спосіб не втратити картину, коли пишеш не ти.

## Три правила greenfield

1. **Harness — перший комміт.** Гейти дешеві, поки нема коду, і безцінні, коли код пише не ти.
2. **Кожен мерж годує harness.** /retro після кожної фічі: розбіжності → правки шаблонів, конвенції → CLAUDE.md, acceptance → трейс. Проєкт, де harness не росте з кодом, за місяць повертається до вайб-кодингу.
3. **YAGNI жорсткіший, ніж без агентів.** Агент генерує архітектуру дешево, а ти її потім читаєш дорого. Абстракція — тільки при другому використанні.
