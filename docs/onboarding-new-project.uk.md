# Онбординг на новий проєкт — покрокова інструкція (harness-підхід)

Сценарій: ти сіньйор, зайшов на чужий проєкт (нова робота / новий клієнт). Кодову базу не знаєш ні ти, ні Claude. Мета перших 30 днів: (1) стати продуктивним швидше, ніж очікує команда, (2) побудувати собі harness, не зламавши командні звички, (3) до кінця місяця мати що запропонувати команді.

Ключова відмінність від власного проєкту: **ти гість.** Тому все ділиться на два рівні:
- **Особисте** (нікого не питаєш): `~/.claude/settings.json`, `CLAUDE.local.md`, локальні skills — діє тільки в тебе.
- **Командне** (тільки після довіри й погодження): комміти в `.claude/` репозиторію, CI-агенти, hooks для всіх.

Перші 2 тижні працюєш ТІЛЬКИ на особистому рівні.

---

## ДЕНЬ 0 (до першої задачі): доступи + запуск

**0.1.** Класика, без якої решта не має сенсу: доступи (repo, CI, трекер, staging, логи/APM), клон, локальний запуск, прогін тестів. Якщо проєкт не піднімається за інструкцією з README — це твоя перша знахідка, запиши.

**0.2.** Заведи особистий журнал онбордингу поза репо:

```bash
mkdir -p ~/onboarding/<project>
touch ~/onboarding/<project>/journal.md   # знахідки, питання, WTF-моменти
touch ~/onboarding/<project>/metrics.md   # та сама таблиця: задача/токени/$/first-pass
```

WTF-моменти — найцінніше: через місяць ти перестанеш їх помічати, а зараз це готовий список для CLAUDE.md і для питань команді.

**0.3.** Постав особисті guard-hooks ЩЕ ДО першої сесії в цьому репо — у `~/.claude/settings.json` (user-level, діє всюди, в git проєкту не потрапляє):

```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "$HOME/.claude/hooks/guard.sh" }] }
    ]
  }
}
```

`~/.claude/hooks/guard.sh` — той самий, що в playbook (exit 2 на `git push --force`, `rm -rf`, `DROP TABLE`, читання `.env`), плюс специфічне для чужого проєкту:

```bash
echo "$CMD" | grep -qE '(prod|production|live)' && deny "команди з prod-контекстом — тільки руками, я тут новенький"
```

На чужому проєкті ціна помилки агента вища, а твоє розуміння "що небезпечно" — нижче. Гейт ставиться до першого ризику, не після.

**0.4.** Перевір, чи в репо вже є `.claude/`, `CLAUDE.md`, `.mcp.json`, `.cursorrules` тощо. Якщо є — прочитай повністю: це командні домовленості про AI-інструменти, їх не ламаєш. Якщо є `.mcp.json` — глянь, які сервери підключаться, і зроби швидкий token-аудит (`/context`): чужі конфіги бувають ненажерливі.

## ДНІ 1–2: розвідка (двосторонній perception gap)

Тут perception-gap аудит працює навпаки: ти перевіряєш не Claude проти свого знання, а **використовуєш Claude як researcher, а людей — як верифікаторів.**

**1.1.** Сесія-розвідка №1 — карта з висоти. Промпт:

```
Досліди цей репозиторій і дай звіт:
1. Що це за система, хто користувачі (з коду/доків, не вигадуй)
2. Архітектура: entry points, шари, як тече запит
3. Стек і версії, зокрема нестандартні/старі залежності
4. Як ганяються тести, скільки їх, що вони покривають на око
5. Топ-10 найбільших/найбільш змінюваних файлів (git log --stat допоможе)
6. Що виглядає як легасі або "тут дракони"
Формат: стислий звіт. Де не впевнений — пиши "ПРИПУЩЕННЯ".
```

Кожне "ПРИПУЩЕННЯ" зі звіту → у journal.md як питання людям.

**1.2.** Сесія-розвідка №2 — потоки грошей/даних. Попроси простежити 2-3 критичні флови end-to-end ("від HTTP-запиту створення замовлення до запису в БД і side effects"). Це найшвидший спосіб знайти справжню бізнес-логіку, а не структуру директорій.

**1.3.** Верифікація людьми. На 1:1 з тімлідом/ментором приходиш не з "розкажіть про проєкт", а з конкретикою:

```
Я вже розібрав: [3 речення]. Перевірте мене в трьох місцях:
1. Правильно розумію, що X працює так?
2. Модуль Y виглядає замороженим — його справді не чіпають?
3. Тести в Z не покривають W — це відомий борг чи я не знайшов?
```

Сіньйора, який на другий день ставить такі питання, запам'ятовують. Відповіді → journal.md.

**1.4.** Побудуй `surface-map.md` — поки що в `~/onboarding/<project>/`, НЕ в репо. Той самий формат: entry points / критичні модулі / стандартні / генерація / зовнішні сервіси / дракони. Це одночасно твій конспект і заготовка майбутнього командного артефакту.

## ДНІ 3–5: особистий CLAUDE.md + перша задача

**2.1.** Створи `CLAUDE.local.md` у корені репо (Claude Code його читає, git його ігнорує — перевір, що він у .gitignore; якщо ні, додай локально через `.git/info/exclude`):

```markdown
# <Project> — мої локальні нотатки (не коммітити)

## Команди (перевірені мною)
- тести: ...       # саме ті інкантації, що спрацювали, з усіма env-змінними
- локальний запуск: ...
- лінт: ...

## Як тут заведено (зі слів команди, дата!)
- 2026-07-06, тімлід: міграції тільки через X, ніколи Y
- 2026-07-07, code review: тут не люблять Z-патерн

## Дракони
- модуль A не чіпати (переписується командою B)

## Мої WTF (перевірити пізніше)
- чому конфіг дублюється в C і D?
```

Правило: у CLAUDE.local.md потрапляє тільки **перевірене** (командою, що спрацювала, або людиною, що підтвердила). Неперевірене живе в journal.md. Інакше ти закодуєш у harness власні ранні помилки.

**2.2.** Візьми першу задачу — свідомо маленьку (good-first-issue, дрібний баг). Мета не "вразити", а **прокатати повний цикл**: гілка → зміна → тести → PR → ревʼю → мерж → деплой. Саме на цьому циклі вилазять усі неписані правила.

**2.3.** Веди задачу spec-lite флоу навіть якщо вона на годину: попроси Claude скласти план на 5 рядків → сам зробив або агент зробив → тести → PR. Задача тут — відкалібрувати, наскільки агенту можна довіряти в ЦЬОМУ репо (у легасі з поганими тестами first-pass rate буде різко нижчий, ніж ти звик у MebleSaaS — заміряй, запиши в metrics.md).

**2.4.** Перше ревʼю твого PR — золото. Кожне зауваження ревʼюера → рядок у CLAUDE.local.md розділ "Як тут заведено". Через 3-4 PR у тебе буде фактичний стайлгайд команди, якого немає в їхніх доках.

## ТИЖДЕНЬ 2: skills + субагенти під цей проєкт

**3.1.** Коли назбиралось 3+ повторюваних процедури — виноси в особисті skills (`~/.claude/skills/<project>-*/`, user-level, не в репо):

- `<project>-testing` — як тут реально пишуть тести (з перших PR і ревʼю)
- `<project>-review` — чеклист самоперевірки перед PR = зауваження, які ти вже ловив
- `<project>-domain` — глосарій домену: що означають терміни в коді (у кожному проєкті є свій "Detail/Item/Position", і плутанина в термінах — половина помилок агента)

Description кожного skill — з конкретними тригер-словами з цього проєкту.

**3.2.** Заведи двох субагентів під онбординг:

- `researcher` — той самий, що в playbook: read-only, звіт ≤400 слів. На новому проєкті це твій головний інструмент — кожна нова задача починається з нього, бо ТВОЄ знання коду ще не покриває dispatch.
- `pr-preflight` — reviewer, натренований на зауваженнях з твоїх перших ревʼю: ганяєш перед кожним PR, щоб людське ревʼю не витрачалось на те, що вже відоме.

**3.3.** Правило економії довіри команди: **жоден PR не йде на людське ревʼю, поки не пройшов pr-preflight.** Твоя репутація в перші тижні = якість перших 10 PR. Агент, що ловить твої ж помилки до ревʼюера, конвертується прямо в неї.

**3.4.** Продовжуй метрики: до кінця тижня 2 у тебе має бути відповідь на питання "на яких типах задач у цьому репо агенту можна делегувати, а де тільки руки" — це твоя особиста dispatch matrix. У легасі-зонах вона буде зовсім інша, ніж у свіжому коді.

## ТИЖДЕНЬ 3: глибина + перший внесок у процеси

**4.1.** Час на глибину в критичних модулях (з surface-map). Техніка: бери researcher-звіт → читай сам ключові файли → пиши в journal.md пояснення "як це працює" своїми словами → проси Claude знайти помилки у твоєму поясненні. Розбіжність = або твоя діра, або знахідка бага.

**4.2.** Задачі бери вже нормального розміру і веди їх повним spec-флоу (spec.md + plan.md локально, не обов'язково коммітити). Бонус: коли на стендапі спитають статус — у тебе є план з кроками, а не "розбираюсь".

**4.3.** Перший процесний внесок команді — обери НАЙДЕШЕВШИЙ для них:
- виправлений README/onboarding-док ("як підняти проєкт за годину, а не за день" — у тебе свіжі очі, зроби це зараз або ніколи), або
- surface-map.md як PR у docs/ — це корисно всім і нічого не нав'язує.

НЕ пропонуй поки hooks/CI-агентів/зміни процесу ревʼю. Зарано.

**4.4.** Якщо в команді вже хтось користується Claude Code / Cursor — знайди цю людину. Це твій канал легалізації harness: удвох пропонувати командний CLAUDE.md сильно легше.

---

## ТИЖДЕНЬ 4: легалізація harness

Тепер у тебе є: перевірений CLAUDE.local.md, особисті skills, метрики за 3 тижні, кілька змержених PR і репутація. Час конвертувати особисте в командне — **через пропозицію, не через факт**.

**5.1.** Підготуй командний `CLAUDE.md` як PR: дистилят твого CLAUDE.local.md мінус особисте, тільки факти й команди, ≤150 рядків. В описі PR — цифри з твого metrics.md ("з цим файлом агент відповідає правильно на N/10 питань про проєкт проти M/10 без нього" — так, зроби perception-gap аудит до/після, тепер класичний, як у playbook).

**5.2.** Якщо PR зайшов — наступними ітераціями (по одному PR, не все одразу): skills → `.claude/settings.json` з fmt-hook (найбезпечніший, нікого не блокує) → guard-hook → далі за rollout-планом з playbook, тижні 2-4.

**5.3.** Якщо команда скептична до AI-інструментів — зупинись на командному CLAUDE.md/доках і продовжуй працювати на особистому рівні. Твій harness працює для тебе незалежно від їхньої згоди; нав'язування зруйнує більше, ніж дасть.

## Три правила, які тримають усе

1. **Перевірене → CLAUDE.local.md, неперевірене → journal.md.** Harness будується тільки з верифікованих фактів.
2. **Особисте одразу, командне — після довіри.** Guard-hooks у день 0, командні hooks — не раніше тижня 4.
3. **Метрики з першого дня.** Через місяць питання буде не "чи допомагає harness", а "покажи цифри" — і в тебе вони будуть.
